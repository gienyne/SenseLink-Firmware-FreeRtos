# FreeRTOS Architecture

This document provides a detailed overview of the FreeRTOS architecture used in **SenseLink**. It explains the design decisions behind the multitasking firmware, including task responsibilities, inter-task communication, synchronization mechanisms, runtime monitoring and memory optimisation.

---

# Architecture Overview

SenseLink is designed around four concurrent FreeRTOS tasks, each responsible for a single subsystem of the firmware.

The architecture follows three key design principles:

* Separation of concerns
* Thread-safe communication
* Deterministic execution on a resource-constrained MCU

The STM32F030R8 provides only **8 KB of RAM**, making efficient task scheduling and memory management essential.

---

# Task Architecture

| Task           |     Stack | Responsibility                                                                           |
| -------------- | --------: | ---------------------------------------------------------------------------------------- |
| **TaskSensor** | 256 words | Reads the BME280 sensor, formats telemetry and distributes data through FreeRTOS queues. |
| **TaskLCD**    | 128 words | Receives formatted strings and updates the HD44780 LCD display.                          |
| **TaskUART**   | 192 words | Sends telemetry over UART and periodically reports FreeRTOS runtime statistics.          |
| **TaskAlarm**  |  64 words | Evaluates alarm thresholds, manages the latching state machine and controls the LEDs.    |
| **IDLE**       | 128 words | FreeRTOS idle task used when no application task is ready to execute.                    |

---

# Task Communication

The firmware avoids global-variable based communication.

Instead, all data is exchanged through typed FreeRTOS queues.

## Queue Creation

```c
UartQueueHandle  = xQueueCreate(2, sizeof(FormattedData_t));
AlarmQueueHandle = xQueueCreate(1, sizeof(SensorData_t));
LcdQueueHandle   = xQueueCreate(3, sizeof(FormattedData_t));
```

## Payload Types

Two message structures are used throughout the firmware.

### SensorData_t

Contains raw floating-point sensor measurements.

Consumed by:

* TaskAlarm

### FormattedData_t

Contains pre-formatted strings generated by `TaskSensor`.

Consumed by:

* TaskLCD
* TaskUART

This design ensures that floating-point formatting occurs only once, reducing stack usage in consumer tasks.

# Queue Design

| Queue      | Depth | Design Rationale                                                             |
| ---------- | ----: | ---------------------------------------------------------------------------- |
| AlarmQueue |     1 | Only the latest measurement is relevant for alarm evaluation.                |
| UARTQueue  |     2 | Absorbs scheduling jitter between producer and consumer.                     |
| LCDQueue   |     3 | Prevents display updates from being delayed by temporary I2C bus contention. |

---

# Shared Resource Protection

Both the BME280 sensor and the HD44780 LCD use the same I2C peripheral.

To prevent concurrent access, every I2C transaction is protected by a FreeRTOS mutex.

```c
osMutexWait(i2cMutexHandle, osWaitForever);
...
osMutexRelease(i2cMutexHandle);
```

The mutex is acquired at the lowest LCD driver level (`LCD_SendCommand()` and `LCD_SendData()`) to avoid recursive locking and potential deadlocks.

Additionally, `TaskSensor` delays its first I2C transaction by three seconds to allow the LCD controller to complete its power-on initialisation sequence.

# Alarm State Machine

The alarm logic is fully isolated inside `TaskAlarm`.

Three operating states are implemented:

* **Nominal**
* **Warning**
* **Critical (Latched)**

# Runtime CPU Monitoring

Every five seconds, `TaskUART` calls `uxTaskGetSystemState()` to collect runtime statistics.

Example output:

```text
--- CPU USE ---
TaskUART     : <1%
TaskLCD      : 0%
IDLE         : 98%
TaskAlarm    : <1%
TaskSensor   : <1%
---------------
```

The collected statistics are transmitted over UART, forwarded through the Python bridge and published to MQTT for real-time visualisation on the dashboard.

# Design Decisions

Several implementation choices were made to maximise reliability on a device with limited resources.

* Typed queues instead of global variables.
* Centralised `snprintf()` formatting.
* Mutex-protected shared I2C bus.
* Interrupt-driven UART reception.
* Runtime CPU monitoring using FreeRTOS APIs.
* Careful tuning of task stack sizes and queue depths.

These decisions allow the firmware to remain deterministic while operating within the memory constraints of the STM32F030R8.
